# スプレッドシート vs OpenAPI.yaml の管理方針

## 現状
- 当初は **スプレッドシート** で API 仕様を管理
  - リクエスト情報・JSONサンプル
  - レスポンス情報・JSONサンプル
- 現在、 **OpenAPI.yaml** がある程度完成
- スプレッドシートの **二重管理が面倒** に感じる

## スプレッドシートを維持するメリット
✅ **一覧性がある** → パラメータやレスポンス項目を俯瞰しやすい  
✅ **非エンジニアと共有しやすい** → YAMLより視認性が良い  
✅ **仕様変更時に比較が容易** → 過去のパターンや検討内容を残せる  

## スプレッドシートをやめるデメリット
❌ **APIの全体像を俯瞰しにくくなる**（ツリー構造のOpenAPIは一覧性が低い）  
❌ **非エンジニアにとってYAMLは読みにくい**（ドキュメント閲覧手段を考慮する必要あり）  

## スプレッドシートをやめるメリット
✅ **二重管理の手間がなくなる**（更新作業がシンプルになる）  
✅ **メンテナンスコスト削減**（API変更時に1つのデータだけ修正すればOK）  
✅ **コードとの整合性を保ちやすい**（API実装とドキュメントのブレを減らせる）  

## 結論
- **OpenAPI.yaml をメインにし、スプレッドシートは廃止してもOK**
- ただし **「エンドポイント一覧 + 概要」だけスプレッドシートに残す** のはアリ
- 非エンジニアとの共有には **Redoc や Swagger UI を活用**
